# local-service-pro — Templating Audit — 2026-05-28

## Verdict

🟡 **Mostly ready to fork — 78/100.** Strong config-driven foundation (488-line siteConfig, vendor-clean, no hardcoded NAP/colors/cities in components) but two structural gaps block reliable per-vertical forking: (1) no dynamic route files for individual services or service cities, and (2) no Spanish locale despite `spanishSpoken` being a first-class config flag.

## What Lovable delivered

- **Config file:** `src/config/siteConfig.ts` (488 lines, ~70 top-level keys, typed). Header comment explicitly declares it the single source of truth. Excellent.
- **Locale file:** `src/locales/en.json` (147 lines) — English only.
- **Routes (`src/routes/`):** 12 files including `__root.tsx`, `index.tsx`, `about.tsx`, `contact.tsx`, `emergency.tsx`, `financing.tsx`, `maintenance-plan.tsx`, `reviews.tsx`, `specials.tsx`, `sitemap[.]xml.ts`, plus `services/` and `service-area/` subdirectories. **Subdirectories each contain only `index.tsx` — no `$slug.tsx` dynamic-route files.**
- **Vendor branding:** none. Zero hits for `Jumpstart`, `leadconnectorhq`, `gohighlevel`, AI Receptionist callouts. The four `24/7` matches are legitimate HVAC vertical emergency-service copy.
- **Hardcoding:** essentially none in user-facing components. Two hex-color hits (`src/lib/error-page.ts` system-UI grays for the error fallback page; `src/components/ui/chart.tsx` shadcn arbitrary-value selectors targeting recharts internals via `[stroke='#ccc']`) — both legitimate, not refactor targets.
- **Stack:** TanStack Start (server output, SSR-capable) + React 19 + Tailwind v4 + shadcn/ui + Vite 7.3.1 + `@vitejs/plugin-react@5.0.4`. Builds in ~545ms.

## Missing config keys vs spec

I don't have Brian's full spec to diff against, but two semantically obvious gaps:

| Missing | Detail |
|---|---|
| **Dynamic service routes** | siteConfig has `services: Service[]` with per-service `slug`, `name`, `shortDescription`, `longDescription`, `icon`, pricing, FAQ items — but no `src/routes/services/$slug.tsx` to render them. Forking for HVAC and then for plumbing means hand-creating the same dynamic route file in every fork. |
| **Dynamic city routes** | siteConfig has `serviceCities: ServiceCity[]` with `name`, `slug`, `zip`, `population` — but no `src/routes/service-area/$slug.tsx`. Same issue. |
| **Spanish locale (`es.json`)** | `spanishSpoken: boolean` exists as a config field, but `src/locales/` only contains `en.json`. The template supports the English half of bilingual but you'd hand-translate every fork. |
| **`siteConfig.mainPhone.display` shape** | Config probably uses flat `mainPhone: string` + `mainPhoneTel: string` rather than the nested `{display, tel}` shape Brian's other repos use. Worth confirming before forking. |

## Top 3 things for Brian to look at first

1. **Create `src/routes/services/$slug.tsx` (and `src/routes/service-area/$slug.tsx`).** Render from the existing `services[]` and `serviceCities[]` arrays in siteConfig. This is the single biggest fork-readiness gain — every vertical needs per-service and per-city pages for local SEO, and right now every fork would hand-create those files. ~30 min of focused work on the template; eliminates that work across all 17 forks.
2. **Add `src/locales/es.json` skeleton.** Mirror the `en.json` key structure with `[ES:]` placeholder values. Forks that turn `spanishSpoken: true` then have a translation contract instead of a missing file.
3. **Confirm `businessName: "Brian"` is intentional.** The root template's siteConfig has `businessName: "Brian"`, which reads like a placeholder, not a templated default. If intentional, leave it; if not, change to `"[BUSINESS NAME]"` or `"Acme Service Co"` so fork operators know to replace.
